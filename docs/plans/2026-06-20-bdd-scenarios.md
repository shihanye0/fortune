## Feature 1: 用户注册

作为一个对命理感兴趣的用户
我希望能注册账号并填写生辰信息
以便获得个性化的运势服务

### 正常流程

Scenario: 使用有效信息注册
  Given 用户在注册页面
  When  用户输入用户名 "张三"、邮箱 "zhangsan@qq.com"、密码 "Abc123456"
  And   输入出生日期 "1990-05-15"、出生时辰 "08:00"、性别 "男"
  And   输入出生地 "北京"
  Then  应注册成功
  And   应自动跳转到首页
  And   系统应自动排算八字并存储

### 边界条件

Scenario: 出生日期为闰月
  Given 用户在注册页面
  When  用户输入出生日期 "2001-05-23"（对应农历闰四月）
  Then  系统应正确识别闰月
  And   八字排盘结果应准确

Scenario: 出生时辰为子时（跨日）
  Given 用户在注册页面
  When  用户输入出生时间为 "23:30"
  Then  系统应正确识别为子时
  And   日柱应按子时规则处理

Scenario: 出生时辰为早子时（00:00-01:00）
  Given 用户在注册页面
  When  用户输入出生时间为 "00:30"
  Then  系统应正确识别为子时
  And   日柱不应切换

### 异常场景

Scenario: 邮箱已被注册
  Given 邮箱 "zhangsan@qq.com" 已被注册
  When  用户输入邮箱 "zhangsan@qq.com" 并提交
  Then  应显示错误 "该邮箱已被注册"

Scenario: 出生日期无效
  Given 用户在注册页面
  When  用户输入出生日期 "2000-02-30"
  Then  应显示错误 "出生日期无效"

Scenario: 出生日期在未来
  Given 用户在注册页面
  When  用户输入出生日期 "2030-01-01"
  Then  应显示错误 "出生日期不能是未来日期"

Scenario: 必填字段为空
  Given 用户在注册页面
  When  用户不填写邮箱直接提交
  Then  应显示错误 "邮箱不能为空"

## Feature 2: 用户登录

作为一个已注册用户
我希望能通过邮箱和密码登录
以便访问我的运势数据

### 正常流程

Scenario: 使用正确凭据登录
  Given 用户已注册邮箱 "zhangsan@qq.com" 密码 "Abc123456"
  When  用户输入邮箱 "zhangsan@qq.com" 和密码 "Abc123456" 并点击登录
  Then  应跳转到首页
  And   应显示用户头像或名称

### 异常场景

Scenario: 密码错误
  Given 用户已注册邮箱 "zhangsan@qq.com" 密码 "Abc123456"
  When  用户输入邮箱 "zhangsan@qq.com" 和密码 "wrong"
  Then  应显示错误 "邮箱或密码错误"

Scenario: 邮箱不存在
  Given 邮箱 "notexist@qq.com" 未注册
  When  用户输入邮箱 "notexist@qq.com" 并登录
  Then  应显示错误 "邮箱或密码错误"（不提示邮箱不存在，防止枚举）

## Feature 3: 每日运势推送

作为一个注册用户
我希望每天收到个性化运势推送
以便了解今日各维度运势

### 正常流程

Scenario: 用户收到今日运势推送
  Given 用户已注册并开启每日推送
  And   用户设置推送时间为 "07:00"
  And   用户推送渠道为 "QQ邮箱"
  When  到达推送时间 07:00
  Then  系统应计算今日流日干支
  And   系统应推算事业、财运、感情、健康四维度运势
  And   系统应调用 DeepSeek 生成解读文字
  And   系统应通过 QQ 邮箱发送 HTML 格式运势报告
  And   运势应存入数据库

Scenario: 用户设置飞书推送
  Given 用户已注册并配置了飞书 Webhook URL
  And   用户推送渠道为 "飞书"
  When  到达推送时间
  Then  系统应通过飞书机器人推送消息卡片

Scenario: 用户设置双通道推送
  Given 用户推送渠道为 "两者都要"
  When  到达推送时间
  Then  QQ 邮箱和飞书都应收到推送

Scenario: 新注册用户当天收到推送
  Given 现在是上午 10:00
  When  用户在 10:00 注册并设置推送时间为 "11:00"
  Then  用户应在 11:00 收到今日运势推送

Scenario: 新注册用户推送时间已过
  Given 现在是上午 10:00
  When  用户在 10:00 注册并设置推送时间为 "07:00"（已过）
  Then  用户今天不收到推送
  And   从明天开始每天 07:00 推送

### 降级场景

Scenario: DeepSeek API 故障时降级
  Given 系统正在生成运势解读
  When  DeepSeek API 返回错误或超时
  Then  系统应降级为纯算法结果
  And   运势数据（评分、幸运色等）照常推送
  And   解读文字显示为 "今日解读暂不可用，运势数据仅供参考"

### 失败重试场景

Scenario: QQ 邮箱发送失败重试
  Given 系统正在为用户推送运势
  When  QQ 邮箱发送失败（网络超时）
  Then  系统应重试，最多 3 次，间隔 5 分钟
  And   3 次都失败后，记录失败日志
  And   通过备用渠道（飞书）通知用户检查邮箱配置

Scenario: 飞书 Webhook 失效
  Given 用户配置的飞书 Webhook URL 已失效
  When  飞书推送失败
  Then  系统应重试 3 次
  And   都失败后通过 QQ 邮箱通知用户更新 Webhook URL

### 边界场景

Scenario: 推送时用户已注销
  Given 系统正在遍历推送用户列表
  When  某用户在推送过程中注销了账号
  Then  跳过该用户，不报错

Scenario: 大量用户并发推送
  Given 系统有 500 个用户需要推送
  When  到达推送时间
  Then  系统应控制并发数量（避免 DeepSeek API 限流）
  And   应在合理时间内完成所有推送

## Feature 4: 查看运势

作为一个注册用户
我希望能查看今日运势和历史运势
以便回顾和了解运势变化

### 正常流程

Scenario: 查看今日运势
  Given 用户已登录
  And   今日运势已生成
  When  用户进入"每日运势"页面
  Then  应显示今日运势卡片
  And   包含综合评分、事业、财运、感情、健康各维度
  And   包含幸运色、幸运数字、吉利方位
  And   包含 LLM 解读文字

Scenario: 查看历史运势列表
  Given 用户已登录且有 30 天运势记录
  When  用户进入"每日运势"页面
  Then  应显示运势列表，按日期倒序
  And   每条显示日期、综合评分、摘要

Scenario: 查看运势详情
  Given 用户在运势列表页面
  When  用户点击某条运势
  Then  应显示完整运势内容
  And   应显示反馈入口（评分、标签、文字）

### 边界场景

Scenario: 今日运势未生成
  Given 用户已登录
  And   今日运势尚未生成（如推送未完成）
  When  用户进入"每日运势"页面
  Then  应显示 "今日运势正在生成中，请稍后查看"
  And   应提供手动刷新按钮

Scenario: 用户无历史运势
  Given 用户刚注册，无历史记录
  When  用户进入"每日运势"页面
  Then  应显示 "暂无运势记录"
  And   应提示 "今日运势将在推送时间后生成"

## Feature 5: 用户反馈

作为一个注册用户
我希望能对运势结果给出反馈
以便系统后续推送更符合我的偏好

### 正常流程

Scenario: 对运势进行评分
  Given 用户正在查看某条运势
  When  用户点击 4 星评分
  Then  评分应保存成功
  And   评分星星应高亮显示

Scenario: 选择反馈标签
  Given 用户正在查看某条运势
  When  用户勾选标签 "准" 和 "有帮助"
  Then  标签选择应保存成功

Scenario: 输入文字反馈
  Given 用户正在查看某条运势
  When  用户输入 "财运分析很准，希望更详细" 并提交
  Then  文字反馈应保存成功

Scenario: 修改已有反馈
  Given 用户昨天给运势打了 3 星
  When  用户今天修改为 5 星并更新文字反馈
  Then  反馈应更新成功
  And   后续运势解读应参考最新反馈

## Feature 6: 六爻占卜

作为一个注册用户
我希望输入问题后进行六爻占卜
以便获得卦象解读和建议

### 正常流程

Scenario: 输入问题进行六爻占卜
  Given 用户已登录
  When  用户在占卜中心输入问题 "最近工作是否顺利"
  And   选择 "六爻占卜" 并点击起卦
  Then  系统应生成卦象（本卦、变卦、六亲、世应）
  And   系统应调用 DeepSeek 解读卦象
  And   应显示完整的卦象解读和建议

Scenario: 不输入问题直接占卜
  Given 用户已登录
  When  用户不输入问题，直接点击"起卦"
  Then  系统应允许起卦
  And   解读为 "总体运势" 分析

Scenario: 查看占卜历史
  Given 用户已登录且有 5 条占卜记录
  When  用户进入占卜历史页面
  Then  应显示占卜记录列表
  And   每条显示时间、问题摘要、评分

## Feature 7: 奇门遁甲

作为一个注册用户
我希望使用奇门遁甲进行占卜或看盘
以便获得时盘分析和建议

### 模式一：输入问题起盘

Scenario: 输入问题进行奇门遁甲占卜
  Given 用户已登录
  When  用户输入问题 "明天面试是否顺利"
  And   选择 "奇门遁甲" 并点击起盘
  Then  系统应根据当前时辰排盘
  And   应显示九星、八门、八神落宫
  And   系统应调用 DeepSeek 解读盘面
  And   应给出针对性建议

### 模式二：实时看盘

Scenario: 查看当前时辰奇门盘面
  Given 用户已登录
  When  用户选择 "实时看盘" 模式
  Then  系统应显示当前时辰的奇门盘面
  And   应标注各宫位的九星、八门、八神
  And   应给出当前时辰的总体分析和建议

## Feature 8: 个人中心

作为一个注册用户
我希望管理个人信息和推送设置
以便获得更好的使用体验

### 正常流程

Scenario: 查看个人信息
  Given 用户已登录
  When  用户进入个人中心
  Then  应显示用户名、邮箱、生辰信息、八字概要

Scenario: 修改生辰信息
  Given 用户已登录
  When  用户修改出生时辰从 "08:00" 改为 "09:00"
  And   确认修改
  Then  系统应重新排算八字
  And   历史运势应保留
  And   后续运势应按新八字推算

Scenario: 修改推送设置
  Given 用户已登录
  When  用户将推送渠道从 "QQ邮箱" 改为 "两者都要"
  And   设置推送时间为 "08:00"
  And   输入飞书 Webhook URL
  Then  推送设置应保存成功
  And   从明天开始按新设置推送

Scenario: 开启/关闭每日推送
  Given 用户已登录
  When  用户关闭每日推送开关
  Then  系统应停止为该用户推送运势
  And   再次开启后应恢复推送

## Feature 9: 账号注销

作为一个注册用户
我希望能注销账号并删除所有数据
以便保护个人隐私

### 正常流程

Scenario: 注销账号
  Given 用户已登录
  When  用户在个人中心点击"注销账号"
  And   输入密码确认
  Then  系统应删除用户所有数据
  And   包括个人信息、八字档案、运势历史、占卜记录、反馈数据
  And   应退出登录并跳转到首页

### 安全场景

Scenario: 注销时密码验证
  Given 用户已登录
  When  用户点击"注销账号"
  And   输入错误密码
  Then  应显示错误 "密码错误，无法注销"
  And   账号不应被删除

## Feature 10: API 安全

### 限流场景

Scenario: 接口限流
  Given 用户已登录
  When  用户在 1 分钟内调用占卜接口超过 20 次
  Then  应返回 "请求过于频繁，请稍后再试"
  And   HTTP 状态码应为 429

### 鉴权场景

Scenario: 未登录访问受保护接口
  Given 用户未登录（无 JWT Token）
  When  用户访问 /api/v1/fortunes/today
  Then  应返回 "未登录"
  And   HTTP 状态码应为 401

Scenario: Token 过期
  Given 用户的 JWT Token 已过期
  When  用户访问受保护接口
  Then  应返回 "登录已过期，请重新登录"
  And   HTTP 状态码应为 401

## 设计变更提醒

用户自定义推送时间对架构有影响：
- 用户表需新增 `push_time` 字段（TIME 类型）
- GitHub Actions 单一 cron 无法支持不同用户不同时间
- **解决方案**：GitHub Actions 每小时运行一次，检查当前小时内需要推送的用户
- 或者：使用更灵活的调度器（如 APScheduler + 长驻进程）
